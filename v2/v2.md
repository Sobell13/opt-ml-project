## 1. Comment on `v2` results

`v2` is much stronger than v1. It contains the full 12-run grid:

```text
SGD:  batch size 64, 512 × learning rates 0.01, 0.05, 0.1
Adam: batch size 64, 512 × learning rates 0.0001, 0.0005, 0.001
```

Each run has 20 epochs and seed 0.

### Best runs overall

Ranked by best test accuracy:

| Rank | Run                        | Best test acc | Best epoch | Final test acc |
| ---: | -------------------------- | ------------: | ---------: | -------------: |
|    1 | `sgd_bs64_lr0.01_seed0`    |    **92.21%** |         17 |         91.90% |
|    2 | `adam_bs64_lr0.001_seed0`  |    **92.14%** |         12 |         91.68% |
|    3 | `adam_bs64_lr0.0005_seed0` |    **92.03%** |         19 |         91.75% |
|    4 | `sgd_bs512_lr0.05_seed0`   |    **91.72%** |         17 |         90.41% |
|    5 | `adam_bs512_lr0.001_seed0` |    **91.56%** |         17 |         91.25% |

The best run is:

```text
SGD, batch size 64, learning rate 0.01
```

with best test accuracy:

```text
92.21%
```

This is a nice result because it shows that SGD is not worse than Adam when the learning rate is tuned.

## 2. Main scientific observations from `v2`

### Observation 1 — small batch performs better than large batch

Best small-batch results:

```text
SGD bs=64:  92.21%
Adam bs=64: 92.14%
```

Best large-batch results:

```text
SGD bs=512:  91.72%
Adam bs=512: 91.56%
```

So in this run, batch size 64 generalizes slightly better than batch size 512.

So small batches may find solutions with better generalization, and later we can test whether those solutions are also flatter using sharpness/Hessian metrics.

---

### Observation 2 — learning rate matters a lot

For SGD with batch size 64:

|   LR | Best test acc |
| ---: | ------------: |
| 0.01 |    **92.21%** |
| 0.05 |        91.46% |
|  0.1 |        89.75% |

So for small-batch SGD, `lr=0.1` is clearly too aggressive. It gives much worse accuracy and higher loss.

For Adam with batch size 64:

|     LR | Best test acc |
| -----: | ------------: |
| 0.0001 |        91.23% |
| 0.0005 |        92.03% |
|  0.001 |    **92.14%** |

Adam is less fragile here, but `0.0001` is probably too slow.

---

### Observation 3 — final checkpoint is often worse than best checkpoint

Several runs peak before epoch 20.

Example:

```text
sgd_bs512_lr0.05_seed0
best test acc:  91.72% at epoch 17
final test acc: 90.41% at epoch 20
```

That is a meaningful drop.

Another example:

```text
sgd_bs64_lr0.01_seed0
best test acc:  92.21% at epoch 17
final test acc: 91.90% at epoch 20
```

This confirms that saving best metrics was the right change.

For the final project, we should probably save a **best checkpoint** as well as the final checkpoint.

---

### Observation 4 — Adam is faster, but not clearly better

Adam with `bs=64, lr=0.001` reaches good performance quickly and has the lowest best test loss:

```text
best test loss = 0.2248
```

But SGD with `bs=64, lr=0.01` obtains the best test accuracy:

```text
92.21%
```

So the fair statement is:

> Adam optimizes quickly and reaches strong performance, but after learning-rate tuning, SGD achieves comparable or slightly better test accuracy in this setting.
